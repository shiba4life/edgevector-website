Edge Vector Foundation

Personal Data Sovereignty
A Vision for Individual Control
Edge Vector Foundation, 2026

People have lost control over their personal data. It is scattered across dozens of platforms, stored in formats they cannot access, governed by terms they did not meaningfully consent to, and exploited in ways they cannot see. The Edge Vector Foundation exists to reverse this.

This paper presents the foundation's vision for personal data sovereignty—an open ecosystem where individuals control how their data is stored, who can access it, and how it is used. We describe six requirements for making this real: seamless ingestion, practical use, security by default, verifiable trustworthiness, long-term viability, and collective value. We then describe how two open-source systems—FoldDB and Exemem—work together to meet these requirements.

1. The Problem

The contemporary digital landscape is defined by a paradox: people generate more personal data than ever before, yet they have less meaningful control over it than at any point in history.

Personal data—medical records, financial documents, photographs, messages, work products, creative output—resides in corporate silos, locked behind proprietary APIs and incompatible formats. Users do not own their data in any practical sense. They cannot move it freely between services. They cannot control who accesses it or how it is used. They cannot verify what has been done with it.

This is not simply a privacy problem. It is an infrastructure problem. The tools people use every day—from email to cloud storage to social media—are built on an architecture where the service provider holds the data and the user holds nothing. When the provider changes terms, gets breached, or shuts down, the user's data goes with it.

Previous attempts to return data ownership to individuals—from personal data stores to self-sovereign identity systems—have struggled with a critical challenge: how to make user-controlled data actually useful. Owning your data in a format no application can read, behind an interface no one can use, is ownership in name only.

The Edge Vector Foundation takes a different approach. Rather than asking people to choose between convenience and control, we are building infrastructure that makes control the more convenient option.

2. The Vision

The foundation's mission is to ensure that people—not corporations or centralized platforms—determine how their data is stored, accessed, and used.

Achieving this requires solving six problems simultaneously. A personal data system that is easy to fill but impossible to use is worthless. One that is useful but insecure is dangerous. One that is secure but cannot be trusted is theater. And one that serves only individuals, with no way to generate shared benefit, will never reach the scale needed to matter.

The six requirements are:

  1. Seamless ingestion. It must be easy to get data in.
  2. Practical use. It must be easy to use the data once it is there.
  3. Security by default. The data must be protected without requiring expertise.
  4. Verifiable trustworthiness. Trust must be proven, not promised.
  5. Long-term viability. The system must work for decades, not just years.
  6. Collective value. Individuals who choose to participate must be able to benefit from the network.

The following sections describe each requirement and how the foundation's technology addresses it.

3. Seamless Ingestion

A personal data store is only as valuable as the data inside it. If getting data in is difficult, people will not do it. The system must support two ingestion paths.

3.1 Self-Ingestion

Users must be able to import their own data—documents, photos, medical records, financial statements, emails, notes—directly from their devices. The system should accept data in whatever format it already exists and organize it automatically.

FoldDB addresses this through a global schema registry: a shared catalog of data formats that every FoldDB instance uses. When new data arrives in an unfamiliar format, an AI-assisted schema generator analyzes the data, checks whether an existing schema can accommodate it, and either maps it to a known format or creates a new one. The result is that data flows in without the user needing to understand schemas, formats, or database design.

Because every FoldDB instance shares the same schema registry, data organized by one person's instance is immediately compatible with every application built on FoldDB. This is the key insight: a global schema registry makes personal data ownership practical by eliminating the fragmentation that has historically made personal data stores unusable.

3.2 Third-Party Writes

Not all personal data originates from the individual. A doctor writes a diagnosis. A bank records a transaction. A colleague shares a document. An application generates usage data.

The system must allow authorized third parties to write data into a person's data store with explicit permission. In FoldDB, this is governed by the same access control system that governs reads: the data owner sets a trust distance and permission policy, and only writers who meet the criteria can add data. Every third-party write is cryptographically signed by the writer, creating an auditable record of who contributed what.

The combination of self-ingestion and permissioned third-party writes means that a person's data store becomes a single, comprehensive record of their information—regardless of where it originated.

4. Practical Use of Personal Data

Ownership is meaningful only if people can actually use their data. A data store that just sits there is a backup drive, not a tool. The system must enable three forms of practical use.

4.1 Applications Operating on Your Data

Software—including AI assistants—must be able to operate directly on a person's data without requiring that data to be uploaded to a centralized service.

FoldDB makes this possible through its universal schema system. Because every FoldDB instance exposes the same schemas through the same API, an application written for one person's data works identically with every other person's data. Developers write code once; it works everywhere. This eliminates the integration tax that has historically made personal data stores impractical for application developers.

AI assistants are one important application of this capability. A personal AI that can query your medical records, financial data, and work documents—all stored locally, all under your control—is far more useful than a cloud AI that sees only what you manually upload. But the architecture is not limited to AI. Any application—a budgeting tool, a health tracker, a project manager—can operate on the user's data through the same standardized interface.

4.2 Selective Disclosure

People regularly need to share specific information with trusted parties: medical records with a doctor, financial documents with an advisor, project files with a collaborator.

In FoldDB, data is never accessed directly. Instead, users interact through folds—named interfaces that sit in front of stored values and enforce access policies on every operation. A fold defines which fields are visible, what transformations are applied, and who may read or write. The same underlying data can have multiple folds: one that shows a doctor everything, another that shows a researcher only de-identified data, and a third that shows a lender only a yes/no creditworthiness answer.

Folds can apply irreversible transforms—such as hashing a name into a de-identified token—so that the recipient gets the information they need without seeing the underlying sensitive data. This is selective disclosure by design, not by policy: the system structurally cannot reveal more than the fold allows.

4.3 Permissioned API Access

External applications must be able to access user data through controlled endpoints with explicit authorization. FoldDB exposes a uniform API across all instances, and every API call passes through the fold's policy checks. An application that the user has authorized can read and write data within the bounds of its permissions; an application without authorization gets nothing—not an error message, not a partial result, nothing.

This creates a practical ecosystem where applications compete on the quality of their service, not on the size of their data moat. Users can switch between applications freely because their data stays in their own store.

5. Security by Default

The system must protect data against unauthorized access while remaining practical to use. Security that requires expertise is security that most people will not have.

5.1 End-to-End Encryption

All data at rest is encrypted. Encryption keys are held by the user, not by any service provider. When data is backed up to the cloud, the cloud stores only ciphertext—encrypted data that is unreadable without the user's key. If the cloud is compromised, the data remains unreadable.

This is not encryption as an add-on feature. It is the foundation of the architecture. The system is designed so that no one other than the user and their explicitly authorized parties can read the data, regardless of where it is stored.

5.2 Granular Access Permissions

FoldDB's access control operates through four independent, conjunctive mechanisms:

  • Trust distance. The data owner assigns a numerical distance to other users. Each data field has a policy specifying the maximum distance for reading and writing. A doctor at distance 1 can see clinical records; a researcher at distance 3 sees only de-identified data; an unauthorized user at distance 10 sees nothing.
  • Cryptographic capabilities. Access tokens tied to specific public keys, with bounded quotas. A capability might grant 100 reads of a specific field, after which access is automatically revoked.
  • Security labels. A classification system that prevents information from flowing from higher to lower security levels through chains of data transformations.
  • Payment gates. Access can require payment, with costs that are a function of trust distance.

All four checks must pass for any data access. Failure at any level produces no output—no partial results, no error messages that reveal structure. This is not a policy promise; it is a structural guarantee enforced by the system's design.

5.3 Economic Safeguards

One of the most novel aspects of the security model is the use of economic mechanisms to prevent large-scale data extraction. Access costs can be configured to increase exponentially with trust distance and with the volume of data accessed. This means that while a trusted doctor can access a patient's records at minimal cost, an adversary attempting to extract data across thousands of users faces costs that grow exponentially—making large-scale harvesting or inference attacks economically impractical, even when individual queries are affordable.

This economic layer complements traditional access control. Even if an attacker obtains valid credentials, the cost structure makes mass extraction prohibitive.

6. Verifiable Trustworthiness

A system that asks people to store their most sensitive data must provide guarantees about what happens to that data. Trust must be proven, not promised.

6.1 Signed Writes

Every data entry—whether written by the owner, a doctor, a bank, or any other authorized party—is cryptographically signed. The signature binds the writer's identity to the data and the timestamp. This means you can verify who wrote what, and when. Tampered data is detectable. Authenticity is provable.

All data is stored in an append-only log: entries are never modified or deleted after insertion. Every previous version of every record is retained, providing a complete, auditable history that supports rollback, dispute resolution, and regulatory compliance.

6.2 Transparent Access Logs

Every read, every read attempt, and every failed access attempt is recorded in an audit log. The user always knows who accessed their data, what they accessed, when they accessed it, and whether the access succeeded or failed. This audit trail is itself append-only and cryptographically signed, making it tamper-evident.

This is a fundamental departure from the current model, where users have no visibility into how their data is being used by the platforms that hold it.

6.3 Data Provenance

When data is transformed—for example, when a medical record is de-identified for research, or when a financial summary is derived from transaction history—the transformation is performed by a registered, vetted function from a shared catalog called the Universal Transform Registry.

Each transform is content-addressed (identified by a hash of its definition), deterministic (same input always produces the same output), and publicly auditable. This means that derived data can be traced to its original source, the transformation that produced it can be independently verified, and the correctness of the output can be validated by anyone.

This chain of provenance—from raw data through signed writes, through registered transforms, to derived outputs—creates a system where trust is not a matter of faith. It is a matter of verifiable cryptographic proof.

7. Long-Term Viability

A personal data system must remain viable for decades. Medical records, financial history, family photographs, and creative work are not disposable. The system must outlast any individual service provider, any particular technology, and any specific device.

7.1 Resilience Through Encrypted Cloud Backup

FoldDB supports encrypted cloud backups that provide redundancy without surrendering sovereignty. The cloud stores ciphertext—encrypted data that is useless without the user's key. If a laptop is lost, the data can be restored. If a cloud provider is compromised, the data remains unreadable. The cloud provides resilience, not access.

This dual architecture—local-first operation with encrypted cloud backup—means the system works offline, survives hardware failures, and does not depend on any single provider for continuity.

7.2 Accessible Permission Management

The access control system described in Section 5 is powerful, but power without usability is dangerous. A non-technical user cannot be expected to manage trust distances, cryptographic capabilities, and security labels manually.

The foundation is building AI-assisted interfaces that help users manage their permissions in plain language. Instead of configuring a fold with W1 R3 trust distance policies, a user says: "Let my doctor see my medical records, but no one else." The AI translates this into the appropriate policy configuration, explains what it means in concrete terms, and asks for confirmation.

Data sovereignty should not require a computer science degree. The system must be as easy to manage as the corporate silos it replaces—and more transparent about what it does.

8. Collective Value from Personal Data

A personal data system that serves only individuals will struggle to reach the scale needed to matter. The real power of data emerges when it can be combined across people—but historically, this has required surrendering it to a central authority. The Edge Vector Foundation's architecture enables collective value without centralized ownership.

8.1 Communal Computation on Anonymized Data

Exemem, the foundation's networked computation layer, enables a fundamentally new form of collective intelligence. When users choose to participate, their data is converted into vector embeddings—mathematical representations that capture the semantic meaning of the data without revealing the specific content. These embeddings are published to a shared network.

The critical privacy mechanism is per-entry derived pseudonyms. Every embedding is tagged with a unique, one-time public key that cannot be linked to the user's identity or to any of their other entries. An observer who sees 1,000 entries sees 1,000 unrelated pseudonyms. They cannot determine whether these belong to 1,000 people or one person. Unlike conventional pseudonymity, where a single alias is reused, this system generates a fresh identity for every data entry.

This enables powerful collective queries. A public health researcher can ask: "How many people match this rare disease profile?" and receive an answer—47 matching entries—without learning who any of them are, whether any two belong to the same patient, or seeing any raw medical data. The embeddings power discovery; the folds protect content.

Remarkably, anonymity in this system strengthens as the network grows. With more users and more entries, the set of people who could plausibly own any given entry becomes larger, making deanonymization harder, not easier. This is the opposite of conventional social networks, where more data means more profiling.

8.2 Interpersonal Sharing

Users can share information with friends, collaborators, or communities in a controlled manner. Because every interaction goes through the fold system, the sharer determines exactly what the recipient sees. You can share travel recommendations without revealing your itinerary. You can share research findings without exposing your raw data. You can share a health insight without disclosing your medical records.

Sharing is a choice, not a requirement. And unlike sharing on a social platform, the data remains in the owner's store—the recipient gets access, not a copy.

8.3 Data Marketplaces

Individuals may choose to allow specific queries against their data and receive compensation when relevant matches are found. The architecture supports this through the combination of public embeddings for discovery and fold-gated access for content.

Consider a scenario: a company is looking for people who match a particular demographic and interest profile for market research. The query runs against the network's public embeddings. Matching entries are discovered. The company can then request access to specific data through the fold system, paying the price set by each data owner. The data owner sees the request, controls what is revealed, and receives compensation.

The user sets the terms. The user controls the access. The user receives the value. This inverts the current model, where platforms harvest user data and sell access to it without the user's knowledge or compensation.

9. How It Works Together

The foundation's vision is realized through two complementary open-source systems.

9.1 FoldDB: The Personal Data Store

FoldDB is the core database. Each user runs their own instance. It provides:

  • A global schema registry that ensures every instance speaks the same data language, making applications universally compatible.
  • Folds—policy-enforcing interfaces that sit between the data and anyone who wants to access it. Folds control what is visible, apply transformations, and enforce trust distance, capabilities, security labels, and payment requirements.
  • Signed, append-only storage where every write is authenticated and every version is retained.
  • A Universal Transform Registry of vetted, deterministic data transformations that enable verifiable data provenance.
  • Encrypted cloud backup for resilience without surrendering control.

FoldDB is where your data lives. It is the sovereignty layer.

9.2 Exemem: The Network Layer

Exemem is the networked computation platform built on top of FoldDB. It provides:

  • Public vector embeddings that enable discovery across the network without exposing raw data.
  • Per-entry derived pseudonyms that make entries unlinkable—no two entries can be connected to the same person.
  • Similarity discovery that lets users find matching data across the entire network.
  • Collective computation—clustering, trend analysis, anomaly detection—on the public embedding layer.
  • A routing layer that connects discovery to access without revealing the mapping between pseudonyms and real identities.

Exemem is how your data creates value beyond your own use. It is the network layer.

9.3 The Two Layers Together

The architecture has a clean separation: Exemem handles discovery (finding relevant data across the network), and FoldDB handles access (controlling what is revealed and to whom). Discovery is public and pseudonymous. Access is private and policy-controlled.

This separation means that the network can grow and become more useful without the privacy risks that growth creates in conventional systems. More participants means better discovery. More participants also means larger anonymity sets, making deanonymization harder. Value and privacy grow together.

10. What This Makes Possible

When these pieces come together, a new class of interactions becomes possible:

Healthcare. A patient's complete medical history lives in their personal data store. Their doctor writes to it directly. A specialist sees only the relevant records. A researcher can discover that 47 patients match a rare disease profile—without seeing any individual records or knowing who any of them are. If a patient wants to participate in a clinical trial, they reveal precisely the information needed, receive compensation, and retain control.

Finance. A person's financial records are in their data store. A lender queries whether they qualify for a mortgage and receives a signed yes/no answer—without seeing bank statements, pay stubs, or tax returns. A financial advisor sees the data they need to give advice. A tax preparer sees exactly what is required for filing, nothing more.

Creative work. A musician stores their compositions. The embeddings are published to the network. If someone publishes a suspiciously similar track, the musician can discover the match, verify the timestamps in the append-only log, and establish a verifiable evidence trail—all without a centralized rights management system.

Personal life. Your photographs, messages, and memories are in your data store. You share what you choose with the people you choose. If someone else has photos of you, you can discover that fact through the network—and request access or removal—without anyone's photo library being browsed or exposed.

In each case, the pattern is the same: the individual controls the data. Access is explicit, granular, and auditable. Collective value emerges from participation, not from surrender.

11. Organizational Structure

The Edge Vector Foundation is a non-profit organization that governs the open-source ecosystem. It maintains FoldDB, the schema registry, the Universal Transform Registry, and the network protocols. Its mandate is the long-term health of personal data infrastructure.

A for-profit subsidiary builds commercial products on top of the open-source foundation: managed cloud backup, enterprise features, mobile applications, and premium support. Revenue funds continued development of the free tools. The subsidiary can never close-source the foundation's work.

This structure ensures that the core infrastructure remains free, open, and community-governed, while creating a sustainable path for long-term development.

12. Conclusion

The current model of personal data management is broken. People's most sensitive information is scattered across corporate silos, accessed without meaningful consent, and exploited without compensation. This is not inevitable. It is a consequence of infrastructure that was built to serve platforms, not people.

The Edge Vector Foundation is building the alternative: open infrastructure where individuals control their data, where access is explicit and verifiable, where security is structural rather than contractual, and where collective value does not require centralized ownership.

The technical foundations exist. FoldDB provides the personal data store with policy-enforcing folds, signed writes, and encrypted storage. Exemem provides the network layer with pseudonymous discovery and collective computation. Together, they form an ecosystem where data sovereignty is not a limitation but a feature—where control and utility reinforce each other rather than competing.

The question is not whether personal data sovereignty is technically possible. It is. The question is whether we build the infrastructure to make it the default.
